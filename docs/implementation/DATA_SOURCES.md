# AIDAN — Data-Source Architecture

How data enters the system, who owns it, and how AI features consume it.

---

## 1. Data categories

| Category | Examples | How it enters | Owner DB / table(s) | Owning service |
|---|---|---|---|---|
| **Scraped** | courses, study programs, degree requirements, appointments, instructors | `services/scraper` (TUMonline XML) | `courses-data`: `courses`, `study_programs`, `program_requirement_areas`, `program_area_courses`, `curriculum_connections`, `course_appointments`, `lectureship`, `persons`, `organizations`, `semesters`, `course_types` | scraper (write); server + genai (read-only) |
| **User-input** | name, study program, semester, interests, career goals, workload pref, expected graduation, industry/role prefs | Onboarding wizard + profile page ONLY | `profiles`: `profiles` table (`student` JSON column) | user-profile-service :8060 (`GET /v1/get/{id}`, `POST /v1/upsert/{id}`) |
| **User-analyzed** | transcript &rarr; completed/enrolled courses | PDF upload &rarr; pdf-parser &rarr; deterministic match &rarr; genai embedding match &rarr; user review | `security`: `student_completed_courses` (status: `confirmed\|pending\|unmatched\|skipped`), `student_enrolled_courses` | server :8080 (`StudentDataDB`) |

Vector embeddings (`course_embeddings` in `courses-data`) are generated by genai from scraped course descriptions. Genai owns the write; server and genai read via vector similarity search.

---

## 2. Iron rules

1. **User-input data is entered in exactly two UIs**: onboarding (`services/client/src/routes/onboarding.tsx`) and profile page (`routes/_authenticated/profile.tsx`). No other form may ask for it.
2. **AI features never take profile fields from the client request.** The server controller fetches the profile (`ExternalServices.fetchProfile()` or direct `restClient` call to profile-service), completed/enrolled courses (`StudentDataDB`), and catalog data (`CoursesDataDB`), then assembles the genai payload server-side. The client sends only feature-specific input (chat text, course IDs, file).
3. **Genai is stateless regarding users.** It receives full context per request; it never stores user data. The only data genai writes is `course_embeddings` in the courses-data DB.

## 4. Known gaps

- **`profiles` table caches stale copies.** The `profiles.completed_courses` / `profiles.enrolled_courses` fields are copies populated by the profile service. The authoritative rows live in `security.student_completed_courses` / `security.student_enrolled_courses`. The recommendations flow reads from the profile cache; other features (advisor, roadmap) read from the security DB directly.
- **Plan validation has no server caller.** GenAI exposes `POST /v1/me/plan/validate` but no Spring Boot controller calls it.
- **Embedding refresh is manual.** After the scraper updates courses, embeddings must be regenerated via the `/v1/embeddings/courses` endpoint. There is no automatic trigger.
